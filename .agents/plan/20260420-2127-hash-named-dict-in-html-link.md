# Fixed Plan: hash-named dict 配信 + h2o include による Link ヘッダ

## 概要

配信用辞書を固定名 `entries.dict` から content-addressed な `<hash>.dict` に切り替える。本文配信時の `Link: <...>; rel="compression-dictionary"` は `h2o.conf.d/dictionary.conf` に build-time で書き出し、`h2o.conf` から静的に読む。HTML に dict URL は埋め込まない。

狙い:

- 辞書ファイル名が content address になるので、クライアントキャッシュと辞書更新タイミングの整合が自然に取れる (同じ URL は常に同じバイト列)
- hash 切替は `make dict` + `h2o reload` で完結し、HTML 再ビルドは不要
- `h2o.conf` は静的 include だけを持ち、`dcb.rb` と `make comp` は唯一の `.dict` を直接参照する

## 現状

- 辞書本体: `blog.jxck.io/dictionary/entries.dict` ([README.md:5](.src/dictionary/README.md#L5))
- h2o 静的 Link ヘッダ: [h2o.conf:318](h2o.conf#L318)
- h2o 辞書 path block: [h2o.conf:306-309](h2o.conf#L306-L309) の `/dictionary/entries.dict`
- h2o 期待 hash 計算: [.mruby.handler/dcb.rb:3-5](.mruby.handler/dcb.rb#L3-L5) で dict body → SHA-256 → `:base64:`
- 辞書生成: [Makefile:250-251](Makefile#L250-L251) で `$(DICT_GENERATOR) -o ./blog.jxck.io/dictionary/entries.dict $(BLOG_HTML)`
- dict-generator.rb は `-d DIR` モードで `<sha256hex>.dict` 出力をサポート済み ([README.md:182-183](.src/dictionary/README.md#L182-L183))
- `.html.dcb` pattern rule は `entries.dict` を固定参照している

## 設計

### 配置

- **辞書実体**: `blog.jxck.io/dictionary/entries/<hash>.dict`
- **配信 URL**: `/dictionary/entries/<hash>.dict`
- **h2o include 断片**: `h2o.conf.d/dictionary.conf`

`entries/` サブディレクトリに寄せることで、将来 `/dictionary/podcasts/<hash>.dict` 等のネームスペース拡張に備える。

### 設計原則

1. **HTML は dict を知らない**: hash 文字列を HTML 出力に入れない → 辞書学習入力の循環依存なし
2. **本文用 Link は静的 include**: `dictionary.conf` は `make dict` が書き出し、`h2o.conf` はそれを静的に読む
3. **辞書ディレクトリには常に 1 本だけ**: `make dict` 冒頭で旧 dict を削除し、新しい hash 名辞書を 1 本だけ生成する
4. **`dcb.rb` と `make comp` は唯一の `.dict` を読む**: `dictionary.conf` を parse しない

### include ファイルの commit 方針

`h2o.conf.d/dictionary.conf` は repo に置き、`make dict` が実 hash URL で上書きする。`make clean` では `header.unset: Link` を書き戻し、`h2o -t -c h2o.conf` が通る状態を保つ。

## 変更内容

### 1. Makefile: dict ターゲット

```make
DICT_DIR  := ./blog.jxck.io/dictionary/entries
DICT_CONF := ./h2o.conf.d/dictionary.conf

$(DICT_CONF): $(BLOG_HTML) ./.src/dictionary/dict-generator.rb
	@rm -f $(DICT_DIR)/*.dict
	@dict_path=$$($(DICT_GENERATOR) -d $(DICT_DIR) $(BLOG_HTML)); \
	  dict_url=$${dict_path#./blog.jxck.io}; \
	  printf "header.set: 'Link: <%s>; rel=\"compression-dictionary\"'\n" "$$dict_url" > $(DICT_CONF)

.PHONY: dict
dict:
	@$(MAKE) -B $(DICT_CONF)
```

- `make dict` は毎回 `$(DICT_CONF)` を更新する
- 旧 `*.dict` は生成前に削除し、`entries/` 配下は常に 1 本だけに保つ
- `dict-generator.rb -d` が返した出力パスから Link ヘッダ値を書き出す

### 2. Makefile: `.html.dcb` pattern rule

```make
%.html.dcb: %.html $(DICT_CONF)
	@dict_path=$$(echo $(DICT_DIR)/*.dict); \
	  ./.src/dictionary/compress.sh \
	    --dict "$$dict_path" \
	    --output-dir . \
	    -dcb \
	    $<
```

`$(DICT_CONF)` を prerequisite にすることで、辞書更新時に `.dcb` が再生成される。辞書パスは `entries/*.dict` の唯一ファイルをそのまま使う。

### 3. Makefile: clean

```make
clean:
	@rm -fv $(COMP_BR)
	@rm -fv $(BLOG_DCB)
	@rm -fv $(DICT_DIR)/*.dict
	@printf '# generated by make dict\nheader.unset: Link\n' > $(DICT_CONF)
```

`$(DICT_CONF)` は `header.unset: Link` に戻す。辞書実体を消した状態でも `h2o -t` が通り、Link が残らないようにする。

### 4. h2o.conf: Link ヘッダを include に差し替え

[h2o.conf:318](h2o.conf#L318) 付近:

```yaml
!file h2o.conf.d/dictionary.conf
```

`"/"` path block の中で、他の `header.set` / `mruby.handler-file` / `file.dir` と同列に置く。

### 5. h2o.conf: 辞書 path block を directory 配信に

[h2o.conf:306-309](h2o.conf#L306-L309):

```yaml
# 旧
"/dictionary/entries.dict":
  header.add: 'Use-As-Dictionary: match="/entries/*", match-dest=("document"), id="entries"'
  header.add: "Cache-Control: max-age=172800"
  file.file: ./blog.jxck.io/dictionary/entries.dict
# 新
"/dictionary/entries/":
  header.add: 'Use-As-Dictionary: match="/entries/*", match-dest=("document"), id="entries"'
  header.add: "Cache-Control: public, max-age=31536000, immutable"
  file.dir: ./blog.jxck.io/dictionary/entries
```

### 6. .mruby.handler/dcb.rb

```ruby

dict_body = dict_path && File.open(dict_path, "rb") { |f| f.read }
```

## Review by Codex

### Findings

1. `DICT_CONF` を placeholder として commit しつつ、`make dict` で実 hash の `header.set` に上書きし、`clean` では触らない設計だと、`make clean` 後の `h2o.conf.d/dictionary.conf` は placeholder に戻りません。結果として `clean` が辞書実体だけ消した後も、h2o は削除済み `<hash>.dict` を指す `Link` を配り続けます。`fresh clone / make clean 後でも h2o -t が通る` という要件は満たせても、`clean` 後の設定整合性は壊れます。`clean` で placeholder に戻すか、`dict-conf-clean` のような固定化した再生成手順を plan に含めるべきです。

2. `設計原則 3` の「reload 前の切替窓で旧 URL を踏む client は 404 になり非圧縮 HTML へ fallback」という説明は、HTTP レベルでは fallback でも、辞書取得自体は 404 で失敗します。`content-addressed URL による整合性` を狙うなら、この 404 窓は避ける設計の方が筋が良いです。少なくとも plan 上は「許容する障害窓」ではなく「運用上の既知トレードオフ」として明記した方が正確です。

### Suggestion

- Jxck さんの指摘どおり、`/dictionary/entries/<hash>.dict` には `Cache-Control: public, max-age=31536000, immutable` を付ける前提を plan に明記した方がよいです。hash 名 URL へ切り替える意味がそこにあります。

## Answer by Claude

Jxck さんの指摘 (Cache-Control) と Codex の 2 findings + suggestion はすべて妥当です。3 点とも採用し、Fixed Plan 本文に直接反映しました。

### 反映内容

**Jxck さん / Codex suggestion — Cache-Control immutable**

`/dictionary/entries/` path block の Cache-Control を `max-age=172800` から `public, max-age=31536000, immutable` に変更しました ([変更内容 5](#5-h2oconf-辞書-path-block-を-directory-配信に))。hash 名 URL が content address である以上、同じ URL は恒久的に同じバイト列であり、`immutable` は正しい宣言です。

**Codex Finding 1 — clean 後の設定整合性**

指摘どおり、既存案では `clean` 後に `dictionary.conf` が古い `<hash>.dict` を指す `header.set` のまま残り、削除済みの辞書 URL を `Link` で広告し続ける不整合がありました。

clean recipe に `@printf '# generated by \`make dict\`\n' > $(DICT_CONF)` を追加し、placeholder 内容へ書き戻すようにしました ([変更内容 3](#3-makefile-clean))。これで `make clean` 直後も `h2o -t` は通り、かつ削除済み URL を指す Link が残らない状態が保たれます。

**Codex Finding 2 — 切替窓の表現**

設計原則 3 の文言を「404 で非圧縮 fallback」から「辞書取得自体は 404 で失敗するが HTML 配信は 200 + 非 dcb で成立する。content-addressed URL + hash rotation に伴う運用上の既知トレードオフとして受容する」に書き換えました ([設計原則 3](#設計原則))。"無害" ではなく "既知トレードオフ" として正確に明記しています。

## Review by Codex

### Findings

1. `.mruby.handler/dcb.rb` が `h2o.conf.d/dictionary.conf` の `header.set` 行を正規表現で読んで active dict を解決する設計になっており、`dictionary.conf` が h2o 専用 include ではなく runtime metadata としても使われています。これは「h2o は静的に読むだけ」という責務分離から外れています。

2. `Makefile` の `.html.dcb` rule も同じ `dictionary.conf` を `sed` で解釈して辞書パスを得る前提になっており、`h2o` / `make comp` / `dcb.rb` の 3 者が同じ設定文字列に依存しています。`dictionary.conf` の記法変更が圧縮生成と配信判定の両方を壊すので、設計上の結合が強すぎます。

3. 現行案には active dict 専用 metadata が存在しないため、結果として `dictionary.conf` を metadata として流用するしかなくなっています。build-time に `dictionary.conf` と別の固定 metadata を同時生成する形に寄せた方が、意図と実装が一致します。

### Suggestion

- `make dict` で `dictionary.conf` とは別に `dictionary.meta` のような固定ファイルを生成し、`h2o.conf` は前者だけ、`dcb.rb` と `make comp` は後者だけを読む構成にした方がよいです。
