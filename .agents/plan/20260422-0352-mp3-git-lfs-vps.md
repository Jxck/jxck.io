### ゴール

`~/server/mozaic.fm/` を新規 git repo として初期化し、そこに mp3 を **Git LFS で管理** できる構成を作る。LFS 実体は VPS (= このサーバ自身) 上の bare repo に保管する。

**範囲外** (今回触らない):

- 既存 `~/server/files.mozaic.fm/` の mp3 を一括で取り込むこと
- `~/server/jxck.io/mozaic.fm/` の markdown 移植や実装の作り直し
- `jxck.io` repo 側の `mozaic.fm/` 削除
- h2o の設定変更 / symlink の張り替え

→ 純粋に「mp3 を LFS で置ける器」を用意するまで。

### 背景 / 前提

- 最終的には `Jxck/mozaic.fm` として GitHub にも push する予定 (今回はまだ)
- mp3 実体は GitHub に上げず、自分だけが保持する (VPS のみ)
- 個人利用 (clone するのは自分のみ) なので最小構成を優先
- サーバは Ubuntu 24.04 LTS。Linuxbrew 導入済みなので `brew install git-lfs` で **git-lfs 3.7.1** が入る (pure-SSH transfer が問題なく使える)
- 既存 `origin` remote が `jxck@jxck.io:/home/jxck/server/jxck.io.git` として動いており、SSH 鍵疎通は実証済み

### 構成

```
VPS (このサーバ) 上:

/home/jxck/server/mozaic.fm.git/         # bare repo (origin + LFS 実体保管)
└── lfs/objects/                         # SHA-256 ハッシュ名で mp3 実体が溜まる

/home/jxck/server/mozaic.fm/             # 作業用 repo
├── .git/
├── .gitattributes                       # *.mp3 filter=lfs
└── .lfsconfig                           # LFS endpoint = ssh://jxck@jxck.io/...
```

LFS のやり取りは pure-SSH transfer (`git-lfs-transfer` プロトコル) を使う。LFS サーバデーモンは不要で、VPS 側に `git-lfs` バイナリが入っていれば SSH 越しに呼び出されて動く。

### 手順

#### 1. git-lfs をインストール

VPS / クライアントとも Homebrew で統一する (既存運用に合わせる)。両サイドで実施:

```bash
brew install git-lfs
git lfs version                     # 3.7.1 が入る想定
```

#### 2. VPS に bare repo を作る

```bash
git init --bare /home/jxck/server/mozaic.fm.git
```

LFS オブジェクトは push 時に `/home/jxck/server/mozaic.fm.git/lfs/objects/` 配下に自動で作られる。事前に何か作る必要はない。

#### 3. 作業用 repo を初期化

```bash
mkdir /home/jxck/server/mozaic.fm
cd /home/jxck/server/mozaic.fm
git init -b main
git lfs install --local
git lfs track "*.mp3"
```

この時点で `.gitattributes` に `*.mp3 filter=lfs diff=lfs merge=lfs -text` が書き込まれる。

#### 4. `.lfsconfig` を VPS に向ける

```bash
cat > .lfsconfig <<'EOF'
[lfs]
  url = ssh://jxck@jxck.io/home/jxck/server/mozaic.fm.git
EOF
```

git-lfs 3.0+ なら `ssh://` URL を指定するだけで pure-SSH transfer が自動で使われる。追加の `standalonetransferagent` 設定は不要。

#### 5. 初回コミット + push で疎通確認

```bash
git remote add origin jxck@jxck.io:/home/jxck/server/mozaic.fm.git
git add .gitattributes .lfsconfig
git commit -m "Initialize LFS-tracked mp3 repo"
git push -u origin main
```

この時点では LFS オブジェクトはゼロなので、純粋に Git 側のみ push される。

#### 6. mp3 1 本で LFS 疎通確認

```bash
cp /home/jxck/server/files.mozaic.fm/mozaic-ep0.mp3 ./
git add mozaic-ep0.mp3
git commit -m "Test: add one mp3 as LFS"
git push
```

確認ポイント:

- `git lfs ls-files` で `mozaic-ep0.mp3` が LFS 管理になっていること
- VPS 側の `/home/jxck/server/mozaic.fm.git/lfs/objects/` に SHA-256 ハッシュ名のファイルができていること
- 別ディレクトリで clone → `git lfs pull` してバイナリが取れること:
  ```bash
  cd /tmp
  git clone jxck@jxck.io:/home/jxck/server/mozaic.fm.git verify
  cd verify
  git lfs pull
  file mozaic-ep0.mp3                 # "Audio file ..." になっていれば成功
  ```

ここまで動けば「mp3 を LFS で置ける構成」は完成。これ以降の作業 (残り 209 本の mp3 取り込み、実装など) は別タスク。

### リスクと対処

- **pure-SSH transfer が期待どおり動かなかった場合**
  - git-lfs のバグ / 設定ミスで `ssh://` URL が受け付けられない可能性はゼロではない
  - その場合はフォールバックとして rudolfs (Rust 製の単機能 LFS HTTP サーバ) を VPS に立てる。ただし h2o リバースプロキシ設定が増えるので、あくまで最終手段
- **LFS による 2 重持ち**
  - LFS では working tree と `.git/lfs/objects/` に同じ実体が置かれる。単純 cp だと二重持ちになるが、手順 6 は 1 本のテスト用なので許容
  - 本格運用時 (210 本取り込み) では `cp -al` (hardlink) を検討する。ただし LFS 内部が hardlink を壊すケースがあるため、実施時に別途検証する
- **バックアップ**
  - `/home/jxck/server/mozaic.fm.git/` は単一障害点になる。既存のバックアップ運用 (rsync / restic 等) の対象に追加することを後続で検討

### 完了条件

- `git-lfs` が VPS とクライアントに入っている
- `/home/jxck/server/mozaic.fm.git/` (bare) が存在する
- `/home/jxck/server/mozaic.fm/` (work tree) が `main` を origin と同期できる
- mp3 1 本を LFS 経由で push / clone / pull できることを確認済み
